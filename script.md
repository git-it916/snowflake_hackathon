## 1. 오프닝 — 문제 정의 (1분)

안녕하세요, 저희 프로젝트 **Telecom Funnel Intelligence**를 소개하겠습니다.

한국 통신사에서 고객이 가입하려면 5단계를 거칩니다.

**상담요청 → 가입신청 → 접수 → 개통 → 납입완료**

각 단계에서 다음 단계로 넘어가는 비율을 **CVR(Conversion Rate, 전환율)**이라고 합니다. 예를 들어 상담요청 100명 중 80명이 가입신청하면 CVR은 80%입니다. 그리고 맨 처음 상담요청 대비 최종 납입완료까지 이어지는 비율을 **OVERALL_CVR, 전체 전환율**이라고 합니다.

문제는 이 과정에서 고객이 많이 빠진다는 겁니다. 통신사는 세 가지를 알고 싶어합니다:

- **어디서** 고객이 빠지는가? → 5단계 중 어떤 구간이 병목인가
- **어떤 채널**이 효과적인가? → 인바운드, 네이버, 카카오 등 38개 채널 중 어디에 돈을 쓸까
- **어디에** 마케팅을 집중해야 하는가? → 200개 시군구 중 어디가 성장하는가

기존에 통신사가 이걸 분석하는 방식은, 엑셀에서 월별 전환율 추이를 보거나, BI 도구로 채널별 매출 바차트를 그리는 정도였습니다. **"어떤 단계를 개선하면 최종 전환율이 정확히 몇 %p 올라가는가?"**, **"전이 확률이 흔들리면 리스크가 얼마나 되는가?"** 같은 질문에는 답할 수 없었습니다.

저희는 이 세 가지 질문에 **데이터로 진단하고, 수학적 모델(마르코프 체인)로 정량화하고, AI Agent로 전략을 자동 생성**하는 시스템을 만들었습니다.

저희 프로젝트의 핵심 엣지를 한 문장으로 말하면:

> **"어디가 문제인가"를 넘어서, "어디를 고치면 얼마의 효과가 있고, 그 예측이 얼마나 불확실한가"까지 정량적으로 답합니다. 그리고 그 답이 정직한지 스스로 검증하는 과정까지 포함했습니다.**

기존 BI 도구는 "개통에서 70% 이탈"이라고 **현황만** 보여줍니다. 저희는 거기서 멈추지 않고 **"그래서 뭘 해야 하는가"**까지 갑니다.

## 2. 데이터 소스 + 전처리 고생담 (1분 30초)

### 데이터 소스

데이터는 **Snowflake Marketplace**에서 가져왔습니다. `SOUTH_KOREA_TELECOM_SUBSCRIPTION_ANALYTICS`라는 데이터셋인데, 실제 한국 통신사의 계약 데이터입니다. 총 5개 뷰를 사용합니다:

- **V03** — 퍼널 전환율 데이터 (250행). 월별 × 카테고리별로 5단계 전환율이 기록되어 있습니다.
- **V04** — 채널별 성과 데이터 (7,936행). 인바운드, 네이버, 카카오 등 38개 채널의 계약 건수, 전환율, 매출이 있습니다.
- **V01** — 지역별 계약 데이터 (23,617행). 서울 강남구, 수원시 같은 200개 시군구 단위입니다.
- **V05** — 지역별 신규 설치 데이터. V01과 합쳐서 번들 가입과 단독 가입을 구분합니다.
- **V07** — GA4 마케팅 어트리뷰션 (8,840행). UTM 소스별 세션과 전환율입니다.

### 전처리 — 예상보다 힘들었습니다

Marketplace 데이터를 그냥 쓰면 안 됩니다. 품질 이슈가 꽤 많았습니다.

**첫 번째로 가장 놀랐던 건**, 전환율이 **347,400%**인 행이 있었습니다. CVR이 100%를 넘을 수가 없는데, 집계 오류로 보입니다. `LEAST(CVR, 100.0)`으로 100% 상한을 걸었습니다.

**두 번째로**, 아직 오지 않은 **2027년 7월** 데이터가 들어 있었습니다. `WHERE YEAR_MONTH <= CURRENT_DATE()`로 미래 날짜를 제외했습니다.

**세 번째가 가장 까다로웠는데**, 퍼널이 **순서대로가 아닌** 경우가 있었습니다. 가입신청 건수가 상담요청보다 많은 카테고리가 있었는데, 상담 없이 온라인에서 바로 가입하는 경로가 존재하기 때문입니다. 이 경우 원래 공식으로 이탈률을 계산하면 **음수**가 나옵니다. 그래서 원본의 이탈률 컬럼을 버리고, 건수 기반으로 직접 재계산했습니다.

**네 번째로**, 현재 월(4월)의 데이터가 아직 수집 중이라 3월의 **1/10 수준**밖에 안 됩니다. 이걸 포함하면 차트가 급락해서 왜곡됩니다. 모든 시계열 분석에서 최신 불완전월을 자동 제외하는 필터를 만들었습니다.

이 외에도 NULL 22% 보정, 도명 불일치('전북특별자치도'→'전북') 통일, 음수 매출(환불 건) 처리 등을 했습니다. 이 전처리가 자동화되어 있고, 매일 **13건의 품질 검증**이 Snowflake Task로 자동 실행됩니다.

여기서도 한 가지 에피소드가 있었는데, ML Feature Store 테이블을 v2로 재생성하면서 테이블 구조가 바뀌었고, 품질 검증 SQL이 **13건 중 6건이 실패**해서 7건만 남는 문제가 발생했습니다. 즉, 파이프라인의 한 테이블을 수정했더니 하류 검증 로직이 깨진 겁니다. 이걸 발견하고 `07_data_quality.sql`을 재실행해서 **13건 전부 PASS**로 복구했습니다. 이런 경험이 Lineage(테이블 의존성 추적)의 필요성을 체감하게 해주었습니다.

## 3. Snowflake 기능 19개 + 개발 방식 (1분 30초)

저희는 Snowflake 기능을 **19개** 사용했습니다. 그리고 이 프로젝트는 **Snowflake가 아니면 만들기 어렵습니다.** 그 이유는, 데이터 저장·전처리·ML 예측·LLM 추론·대시보드 배포가 **전부 하나의 플랫폼 안에서** 이루어지기 때문입니다. 다른 환경이었다면 AWS S3 + SageMaker + OpenAI API + Vercel 등 4~5개 서비스를 연동해야 했을 겁니다. Snowflake에서는 SQL과 Python만으로 End-to-End를 완성했습니다.

개발은 **로컬에서 테스트하고, Snowflake에 배포**하는 방식입니다.

크게 네 가지 레이어입니다.

**첫 번째, 데이터 파이프라인**입니다. SQL 파일 11개를 Snowflake 워크시트에서 순서대로 실행하면, Marketplace 원본 → Staging(전처리) → Analytics(분석) → Mart(대시보드) 순으로 데이터가 흐릅니다. 즉, 원본 데이터를 깨끗하게 씻고, 분석 지표를 계산하고, 대시보드에서 바로 볼 수 있는 형태로 만드는 3단계 과정입니다. **Dynamic Tables**로 1시간마다 자동 갱신하고 — 즉, 원본이 바뀌면 분석 테이블도 알아서 업데이트되고 — **Tasks**로 매일 06시 품질 검사를 실행하고, **Alerts**로 문제 발견 시 알림을 보냅니다.

**두 번째, ML 레이어**입니다. 잠시 후 자세히 설명하겠습니다.

**세 번째, Cortex AI**입니다. Snowflake 서버 안에서 돌아가는 AI 기능 7종을 사용했습니다: **FORECAST**로 3개월 예측 — 즉, "다음 분기에 서울에서 계약이 몇 건 나올까?" 같은 질문에 자동으로 답하는 기능입니다. **ANOMALY**로 이상 탐지 — 즉, 갑자기 계약이 급증하거나 급감한 지역을 자동으로 찾아줍니다. **SENTIMENT**로 감성 점수, **CLASSIFY_TEXT**로 채널 등급 분류, **SUMMARIZE**로 한 줄 요약, **TRANSLATE**로 한영 번역, 그리고 **COMPLETE**(llama3.1-405b)로 Multi-Agent 시스템을 구동합니다. 즉, Snowflake 안에 405억 개의 파라미터를 가진 대형 AI 모델이 내장되어 있어서, **데이터를 외부로 보내지 않고도** AI 분석이 가능합니다.

**네 번째, 거버넌스**입니다. 13건 품질 자동 검증, 테이블 의존성 자동 추적(Lineage) — 즉, "이 테이블을 바꾸면 어떤 대시보드에 영향이 가는지" 한눈에 보는 기능입니다 — 그리고 **Cortex Analyst**를 적용했습니다. Cortex Analyst는 "이번 달 인터넷 채널별 매출 알려줘"처럼 한국어로 질문하면 자동으로 SQL을 만들어서 답을 주는 기능입니다.

대시보드는 로컬에서 Streamlit으로 개발·테스트한 뒤, `deploy_sis.py` 한 번으로 **Streamlit in Snowflake(SiS)**에 배포합니다. 즉, 제 컴퓨터가 꺼져 있어도 Snowflake 안에서 대시보드가 계속 돌아가고, 심사위원분들도 URL 하나로 바로 접속할 수 있습니다.

CoCo 스킬은 5개 영역을 커버했습니다: AI 함수 활용, 머신러닝 파이프라인, 데이터 품질 관리, 테이블 의존성 추적, 비용 최적화입니다.

## 4. ML 모델 — 쉽게 설명 + 3단계 개선 스토리 (1분 30초)

ML 모델이 하는 일을 한 마디로 말하면:

> **"다음 달에 이 카테고리의 전환율이 높을까, 보통일까, 낮을까?"**

학습 과정을 쉽게 풀면 이렇습니다:

1. 채널 성과 데이터에서 카테고리별로 **특성값(피처)**을 만듭니다. 예를 들어 "지난달 전환율은 몇 %였나", "최근 3개월 추세가 올라가는가 내려가는가", "특정 채널에 너무 쏠려 있지는 않은가" 같은 것들입니다. 즉, 사람이 채널 성과를 판단할 때 보는 관점들을 숫자로 만든 겁니다.
2. 그 다음 달 실제 전환율을 보고 **"상위 33%면 HIGH, 하위 33%면 LOW"** 라벨을 붙입니다. 즉, 정답지를 만드는 겁니다.
3. 이 데이터로 **XGBoost**라는 모델을 학습시킵니다. 즉, "이런 패턴이면 다음 달은 HIGH다", "이런 패턴이면 LOW다"를 과거 데이터에서 규칙을 찾아내는 겁니다.

이 과정에서 **3번의 큰 문제와 개선**이 있었습니다.

### 1차 문제: 피처 9개 누락 (F1 0.53)

처음에 Feature Store를 Python으로 만들었더니, 모델이 기대하는 피처 중 **9개가 누락**되었습니다. 즉, 시험 범위의 1/3을 안 알려주고 시험을 보게 한 셈입니다. Python 코드가 SQL로 만든 Feature Store를 **덮어써버리는** 버그가 있었습니다. 이걸 수정해서 SQL Feature Store를 직접 사용하게 했더니, F1이 **0.53에서 0.93으로** 올랐습니다.

### 2차 문제: 0.93은 가짜였습니다

그런데 기뻐하기엔 일렀습니다. 분석해보니 **심각한 문제가 3가지** 있었습니다:

**첫째, 데이터 누수(Data Leakage)가 있었습니다.** 피처 중 FUNNEL_OVERALL_CVR이라는 것이 있었는데, 이게 당월 전환율(PAYEND_CVR)과 **상관계수 1.0**, 즉 완전히 같은 값이었습니다. "이번 달 전환율을 보고 다음 달 전환율을 예측"하는 건 거의 답지를 보고 시험 보는 것과 같습니다. 마찬가지로 FUNNEL_TOTAL_COUNT도 CONTRACT_COUNT와 동일했습니다.

**둘째, 22개 피처 중에서 상관계수 0.95 이상인 쌍이 24개나 되었습니다.** 즉, 22개 피처가 실제로는 **5~6개의 독립적인 정보**만 담고 있었습니다. 예를 들어 "지난달 전환율", "3개월 평균 전환율", "6개월 평균 전환율"은 거의 같은 숫자입니다.

**셋째, 교차검증(CV) 점수가 0.43으로 매우 낮았습니다.** In-sample 0.93 vs CV 0.43 — 이 0.5 차이는 전형적인 **과적합** 신호입니다. 즉, 학습 데이터는 잘 맞추지만 새로운 데이터에는 못 맞추는 상태입니다.

### 3차 수정: 정직한 모델 (F1 0.74, AUC 0.90) — 💡 엣지 포인트

이걸 해결하기 위해 세 가지를 수정했습니다:

1. **데이터 누수 피처 제거** — 당월 전환율과 동일한 FUNNEL_OVERALL_CVR, FUNNEL_TOTAL_COUNT 삭제
2. **중복 피처 정리** — 22개에서 **12개 독립 피처**로 축소. 상관 0.95+ 쌍이 24개에서 **1개**로 줄었습니다.
3. **모델 복잡도 축소** — 트리 깊이 4→2, 트리 수 200→50, 정규화 강화. 즉, 모델이 데이터를 너무 외우지 못하게 제한한 겁니다.

결과: In-sample F1 **0.74**, CV F1 **0.47**, AUC **0.90**. Gap이 0.50에서 **0.27**로 줄었습니다. 0.93보다 숫자는 낮지만, 이게 **정직한 성능**입니다. 151행이라는 소규모 데이터에서 AUC 0.90이면 실용적 가치가 있습니다.

보통 해커톤에서는 F1 0.93을 자랑합니다. **저희는 일부러 내렸습니다.** "높은 점수를 스스로 의심하고, 데이터 누수를 발견하고, 정직한 모델로 수정한다" — 이 과정 자체가 ML 전문성을 보여주는 가장 강력한 방법이라고 생각합니다.

학습된 모델은 **Snowflake Model Registry**에 버전별로 저장됩니다. 즉, v1, v2처럼 모델이 업데이트될 때마다 이전 버전으로 돌아갈 수 있습니다.

여기서 한 가지 더 — 모델이 "LOW"라고 판단했을 때, **"왜 LOW인가?"**를 설명할 수 있어야 합니다. 그래야 실무에서 신뢰를 받으니까요. 이걸 해주는 게 **SHAP**이라는 해석 도구입니다. 즉, XGBoost라는 블랙박스를 열어서 "12개 피처 중 어떤 것이 이 예측에 가장 크게 기여했는가"를 숫자로 분해해줍니다.

그런데 이것도 **한 번에 되지 않았습니다.** Snowpark ML로 학습한 XGBoost 모델에서 내부 구조를 꺼내는 과정에서 배열 차원 에러가 발생했습니다. 최신 SHAP 버전과 Snowpark ML 래퍼의 호환 문제였습니다. 이걸 해결하기 위해 sklearn XGBoost로 동일 데이터를 재학습하는 폴백을 만들고, SHAP 값 변환에서 numpy 배열을 명시적으로 평탄화하는 처리를 추가했습니다.

결과적으로 **실제 SHAP 분석이 작동**하고, 1위 피처는 "지난달 전환율"(기여도 0.40), 2위는 "월(계절 효과)"(0.29), 3위는 "3개월 전 전환율"(0.24)이 나옵니다. 즉, "직전 달 성과"가 다음 달 예측에 가장 중요하다는, 직관적으로도 납득이 가는 결과입니다.

### 진단 페이지 (1분)

왼쪽의 **퍼널 차트**를 보시면 상담요청부터 납입완료까지 단계별 건수가 깔때기 형태로 줄어듭니다. 오른쪽에 빨간 글씨로 **이탈 건수와 이탈률**이 표시됩니다. 어디서 고객이 가장 많이 빠지는지 한눈에 보입니다.

오른쪽의 **버블 차트**는 38개 채널을 비교합니다. X축이 계약 건수, Y축이 전환율이고, 각 버블에 채널 이름 라벨이 달려 있습니다. 초록=성장, 빨강=쇠퇴입니다.

아래로 내려가면 **전환율 추이 차트**가 있습니다. 여기서 **Cortex ANOMALY**가 탐지한 이상치가 X 표시로 나타납니다.

왼쪽 마르코프 — 여기가 **저희 프로젝트의 가장 큰 엣지**입니다. 단순히 "개통에서 70% 이탈"이라는 현황을 넘어서, **"접수→개통을 5%p 올리면 최종 전환율이 +4.5%p, 월 +450건 추가 전환"**이라는 정량 답을 줍니다. 즉, 마케팅 예산을 어디에 쓸지 수학적 근거를 제공하는 겁니다. 이건 엑셀이나 일반 BI 도구에서는 불가능합니다.

오른쪽 STL에서는 추세(하락/상승/안정)와 계절성 강도를 보여줍니다.

맨 아래 **3개 인사이트 카드**가 있습니다. 첫째 카드는 **병목 구간** — 실제 데이터에서 계산된 최악 이탈 단계와 이탈률입니다. 둘째는 **채널 추천** — 효율이 가장 높은 채널입니다. 셋째는 **시즌 패턴** — STL 분해에서 나온 추세와 계절성 강도를 보여줍니다. 이 세 번째 카드가 처음에는 "이사 시즌에 25% 증가"라고 하드코딩되어 있었는데, 검증 과정에서 이게 **데이터가 아니라 텍스트를 박아넣은 가짜**라는 걸 발견하고, STL 실제 결과로 연결했습니다.

그 아래 왼쪽에 **지역별 수요 점수** 수평 바차트가 있습니다. Z-score로 정규화했습니다. 즉, 전국 평균과 비교해서 양수면 수요가 강한 지역입니다. 오른쪽에는 **성장률 상위 도시 TOP 10**이 있습니다.

그리고 **마르코프 전이 확률 시뮬레이션** — 슬라이더로 각 단계 전이 확률을 조정하면 최종 전환율이 즉시 재계산됩니다. 프리셋 시나리오 버튼도 있습니다.

마지막으로 **Monte Carlo 500회** — 이것도 **저희만의 엣지**입니다. 민감도 분석이 "이론적 최적"이라면, Monte Carlo는 **"현실의 불확실성"**을 보여줍니다. "개통 전환율을 올리면 좋겠지만, 매달 ±3%p씩 흔들린다면 최종 전환율이 최악의 경우 몇 %, 최선의 경우 몇 %인가?" — 이 **리스크 정량화**는 경영진이 투자 결정할 때 진짜 필요한 정보인데, 대부분의 프로젝트에서 안 합니다. 히스토그램과 박스플롯, 리스크 수준 게이지로 시각화됩니다.

### AI 전략 페이지 (1분)

*(화면이 좌우 6:4 분할)*

각 Agent의 LLM 추론은 전부 **Snowflake 서버 안에서** 실행됩니다.

결과는 **3개 탭** — "데이터 분석", "채널 전략", "경영진 요약"으로 나뉩니다. 분석 실행 전에도 마르코프 Steady State와 민감도 결과가 즉시 표시되어, 데이터 기반 인사이트를 바로 볼 수 있습니다.

### 향후 전망과 확장성

현재 한계를 솔직히 말씀드리면, ML 모델이 학습한 데이터가 150행으로 적습니다. 즉, 5개 카테고리 × 30개월치밖에 안 되다 보니, 새로운 패턴에 대한 예측이 불안정할 수 있습니다. 데이터가 더 쌓이면 정확도가 올라갈 겁니다.

하지만 **확장성 측면에서 이 구조의 장점**이 있습니다. Snowflake의 AI 모델이 업그레이드되면 — 예를 들어 llama3.1이 llama4로 바뀌면 — 저희 코드에서 **모델명 한 줄만 바꾸면** Agent 시스템 전체가 업그레이드됩니다. 즉, AI가 발전하면 저희 시스템도 같이 좋아지는 구조입니다. ML 모델도 버전별로 저장되어 있어서, 새 버전이 이상하면 이전 버전으로 돌아갈 수 있습니다.

**비용 측면**에서는, 데이터 갱신 주기를 1시간으로 설정해서 불필요한 실시간 갱신을 방지했습니다. 즉, 초 단위 갱신이 필요 없는 비즈니스인데 실시간으로 돌리면 비용 낭비니까요. Cortex FORECAST는 서버를 따로 띄울 필요 없이 SQL 한 줄로 실행됩니다. 그리고 데이터가 10배로 늘어나도 Snowflake가 알아서 서버를 키워주기 때문에, 코드를 수정하지 않고도 운영 가능합니다.

### "F1이 0.74면 좋은 건가요?"

먼저, 처음에는 0.93이었는데 일부러 내린 겁니다. 0.93일 때 분석해보니 데이터 누수(상관 1.0인 피처 2개)와 다중공선성(상관 0.95+ 쌍 24개)이 있었습니다. 즉, 답지를 보고 시험 본 셈이었습니다. 이걸 정리하고 나니 0.74가 됐는데, 이게 정직한 성능입니다. CV F1은 0.47로 151행 소규모 데이터의 한계이지만, AUC 0.90이면 실무에서 의사결정 보조 도구로 충분합니다. 데이터가 쌓일수록 성능이 올라갈 구조입니다.

### "데이터 누수가 뭔가요?"

쉽게 말하면, 모델에게 "이번 달 전환율"을 알려주고 "다음 달 전환율"을 맞추라고 한 겁니다. 이번 달과 다음 달 전환율은 상관이 0.89나 되니까, 거의 답을 보여준 셈입니다. FUNNEL_OVERALL_CVR이라는 피처가 PAYEND_CVR과 완전히 같은 값(상관 1.0)이었는데, 이걸 발견하고 제거했습니다.

### "SHAP이 뭔가요?"

XGBoost 모델이 "LOW"라고 예측했을 때, "왜?"라는 질문에 답해주는 도구입니다. 12개 피처 각각이 이 예측에 얼마나 기여했는지를 숫자로 분해합니다. 예를 들어 "지난달 전환율이 낮아서 +0.40, 3월이라 계절 효과 +0.29" 이런 식입니다. 실제 결과에서 1위가 "지난달 전환율"(0.40)인데, 직전 달 성과가 다음 달에 가장 큰 영향을 준다는 건 직관적으로도 맞습니다. 처음에는 Snowpark ML과 SHAP 호환 문제로 에러가 났지만, sklearn 폴백과 numpy 배열 처리를 수정해서 해결했습니다.

### "마르코프 체인이 뭔가요? 쉽게요."

"현재 퍼널 구조가 그대로 유지된다고 가정하면, 고객 100명이 들어왔을 때 최종적으로 몇 명이 납입까지 가는가?"를 계산하는 수학 모델입니다. 그리고 "어떤 단계를 살짝 개선하면 최종 결과가 얼마나 바뀌는가?"를 정량적으로 알 수 있습니다.

### "로컬과 Snowflake의 관계가 뭔가요?"

SQL과 Cortex AI는 Snowflake 서버에서 실행되고, ML 학습·마르코프 체인·차트 시각화는 로컬 Python에서 실행됩니다. **Snowpark**이라는 Python SDK로 Snowflake 데이터를 가져옵니다. 개발 완료 후 `deploy_sis.py`로 Snowflake 안에 배포하면, 인증 없이 Snowsight에서 바로 접속 가능합니다.

### "13건 품질 검증은?"

NULL 3건 + CVR 범위 2건 + 미래날짜 3건 + 행수 4건 + 중복 1건 = **총 13건**. Snowflake Task가 매일 06시에 자동 실행하고, CRITICAL이면 Alert가 발동합니다.
